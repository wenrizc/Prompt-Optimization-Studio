# 当前项目提示词清单与作用说明

## 1. 先说明边界

基于当前仓库代码，项目里并没有内置一批已经落库的业务 Prompt 数据：

- 当前仓库没有种子数据或初始化 SQL 向 `prompts` 表插入记录
- 当前本地也没有 `data/app.db`

因此，所谓“当前项目中的提示词都有什么”，更准确地说，应该分成两类：

1. 用户在产品里可创建、可编辑、可版本化的业务 Prompt
2. 系统内部为了生成数据、评测、优化、生成摘要而写死在代码里的系统提示词

这份文档重点介绍第二类和第一类的默认模板，因为它们是当前代码中真实存在、可以确认的 Prompt。

## 2. 项目里 Prompt 的总体分类

当前项目里的提示词大致有 7 类：

1. 业务 Prompt：用户在 Prompt Studio 页面编辑的 `system_prompt` 和 `user_template`
2. DSPy 运行包装 Prompt：系统把业务 Prompt 包成 DSPy 指令时生成的总指令
3. 合成数据生成 Prompt：系统调用 LLM 自动生成 synthetic dataset 时使用
4. 自定义任务模板草稿生成 Prompt：系统根据自然语言生成任务模板草稿时使用
5. LLM Judge 评测 Prompt：`llm_judge` 指标打分时使用
6. 报告摘要生成 Prompt：为 evaluation / optimization 生成 executive summary 时使用
7. 优化结果附加 Prompt：优化完成后，系统给 `system_prompt` 自动附加的说明文字

下面分别解释。

## 3. 用户可编辑的业务 Prompt

## 3.1 结构

业务 Prompt 存在 `Prompt` 模型中，由两部分组成：

- `system_prompt`
- `user_template`

字段定义在：

- [backend/models/prompt.py](D:\project\Prompt-Optimization-Studio\backend\models\prompt.py)
- [backend/schemas/prompt.py](D:\project\Prompt-Optimization-Studio\backend\schemas\prompt.py)

它们的作用分别是：

- `system_prompt`：定义模型的角色、输出风格、行为约束
- `user_template`：定义每条输入样本如何组织成实际发给模型的文本

## 3.2 当前前端内置默认值

Prompt Studio 页面在没有选中已有 Prompt 时，会给出一组默认模板。

英文默认值：

- `system_prompt`
  - `You are a precise task assistant.`
- `user_template`
  - `Classify the following request:\n{text}`

中文默认值：

- `system_prompt`
  - `你是一个精准的任务助手。`
- `user_template`
  - `请对下面请求进行分类：\n{text}`

来源：

- [frontend/lib/i18n/messages.ts](D:\project\Prompt-Optimization-Studio\frontend\lib\i18n\messages.ts)
- [frontend/app/[lang]/prompts/page.tsx](D:\project\Prompt-Optimization-Studio\frontend\app\[lang]\prompts\page.tsx)

## 3.3 这些业务 Prompt 有什么用

这是整个系统里最核心的一类 Prompt。

它们会被用于：

1. 创建 Prompt 版本
2. 发起 baseline evaluation
3. 发起 optimization run
4. 被冻结进 `prompt_snapshot_json`
5. 被转换成 DSPy 可执行指令
6. 在优化后生成新的 `optimized_prompt`

换句话说，产品里“真正要优化的对象”就是这组业务 Prompt。

## 3.4 业务 Prompt 的约束

`user_template` 不是随便写的，当前代码有明确校验规则：

1. 必须包含 `{text}`
2. 不能出现未知变量
3. 过长会给 warning

具体规则来自：

- [backend/schemas/prompt.py](D:\project\Prompt-Optimization-Studio\backend\schemas\prompt.py)
- [backend/services/validators.py](D:\project\Prompt-Optimization-Studio\backend\services\validators.py)

这说明当前系统默认的输入变量模型比较简单：

- 运行时主要围绕单个 `text` 字段展开
- `user_template` 本质是把样本 `input_json.text` 渲染到模板中

## 4. DSPy 运行包装 Prompt

## 4.1 具体内容

当系统真正执行 evaluation 或 optimization 时，并不是把 `system_prompt` 和 `user_template` 原样直接扔给 DSPy。

它会先在 `build_instructions()` 中拼成一个总 Prompt，大意如下：

- 你正在执行一个 prompt optimization task
- 只输出最终答案
- 插入业务 `system_prompt`
- 插入业务 `user_template`
- 插入 `output_schema_json`
- 强制要求输出严格匹配 schema 的 JSON，不允许 markdown，不允许解释文字

来源：

- [backend/services/dspy_program_factory.py](D:\project\Prompt-Optimization-Studio\backend\services\dspy_program_factory.py)

## 4.2 它的作用

这个包装 Prompt 是系统执行层真正发给 DSPy 的核心指令。

它的作用是把三种信息合并：

1. 业务意图：来自用户写的 `system_prompt`
2. 输入组织方式：来自 `user_template`
3. 输出契约：来自 `output_schema_json`

最终形成一个“既知道做什么，又知道怎么输出”的运行时 Prompt。

## 4.3 为什么要这样设计

如果只把用户写的 Prompt 直接丢给模型，会有两个问题：

1. 输出格式不稳定
2. 很难保证和项目 schema 严格对齐

所以当前项目采用了“业务 Prompt + 平台包装约束”的模式。

这意味着：

- 用户负责任务表达
- 平台负责输出规约和结构化约束

## 5. 合成数据生成 Prompt

## 5.1 指令 Prompt

在生成 synthetic dataset 时，系统会构造一条内部 Prompt：

- 要求为 prompt optimization project 生成 synthetic dataset examples
- 要求贴合当前 task
- 要求样例真实、多样、简洁
- 要求 `expected_output` 严格满足声明的 schema

来源：

- [backend/services/dataset_generation.py](D:\project\Prompt-Optimization-Studio\backend\services\dataset_generation.py)

## 5.2 输入 Prompt

除了上面的系统说明，它还会拼接一段包含上下文的输入文本，内容包括：

- `Task key`
- `Task description`
- `Command`
- `Count`
- `Input schema`
- `Output schema`
- 多样性要求
- 顶层 JSON 必须是 `items`
- 一个 few-shot JSON 结构示例

## 5.3 它的作用

这类 Prompt 不用于业务预测，而用于“造数据”。

它的目标是让模型返回一批结构化训练/测试样本，每条样本都包含：

- `text`
- `expected_output`

换句话说，它是一个“数据工厂 Prompt”。

## 5.4 为什么重要

Prompt Optimization Studio 的很多实验都依赖数据集。

这个生成 Prompt 的作用是：

- 在没有现成数据时快速起盘
- 让生成数据尽量和项目任务契约一致
- 尽量减少生成结果和 schema 不一致的问题

## 6. 自定义任务模板草稿生成 Prompt

## 6.1 指令 Prompt

当前项目支持用户输入自然语言，让模型帮忙生成一份自定义任务模板草稿。

对应的系统提示大意是：

- 你要为 prompt optimization studio 生成 custom task template draft
- 返回一份最小但可用的模板
- `task_key`、输出字段、metric field、`target_field`、`primary_output_field` 必须内部一致
- 优先使用顶层 object schema
- 不要输出 markdown
- 说明文字要解释每项配置如何使用，哪些字段可改

来源：

- [backend/services/custom_task_template_generator.py](D:\project\Prompt-Optimization-Studio\backend\services\custom_task_template_generator.py)

## 6.2 输入 Prompt

输入部分会进一步要求模型必须返回：

- `draft.task_key`
- `draft.task_display_name`
- `draft.task_description`
- `draft.input_schema_json`
- `draft.output_schema_json`
- `draft.default_metric_config_json`
- `draft.task_definition_json`
- `draft.report_profile_json`
- `guidance.overview`
- `guidance.items`

还特别要求：

- 用户可见说明文字使用中文
- 顶层 JSON key 必须是 `draft` 和 `guidance`

## 6.3 它的作用

这类 Prompt 不是直接优化 Prompt，也不是做评测。

它的作用是：

- 帮用户更快创建“任务契约”
- 降低自定义任务配置门槛
- 保证生成的 schema、metric、report profile 彼此对齐

它是“配置生成器 Prompt”。

## 7. LLM Judge 评测 Prompt

## 7.1 指令 Prompt

当项目的 metric 选择 `llm_judge` 时，系统会构造一条 Judge Prompt。

它的系统指令大意是：

- 你要判断模型预测是否满足 gold answer
- 返回结构化 judgment
- 必须包含 `correct`、`score`、`rationale`、`error_type`、`confidence`

来源：

- [backend/services/metric_factory.py](D:\project\Prompt-Optimization-Studio\backend\services\metric_factory.py)

## 7.2 输入 Prompt

输入部分会附带：

- 当前 metric field
- `Expected output`
- `Model prediction`
- 要求从语义正确性、风格一致性、schema 对齐角度判断
- 一段示例 JSON 响应格式

## 7.3 它的作用

这条 Prompt 的作用是把“评分”本身交给另一个 LLM。

适合：

- 语义判断比较复杂
- 不能简单靠 exact match/F1 判断
- 需要模型辅助给出理由和错误类型

## 7.4 它的特点

和前面的业务 Prompt 不同，这里它不是“被优化对象”，而是“评委 Prompt”。

如果没配置 OpenAI Key，系统还会回退成 heuristic judge，不走这条 Prompt。

## 8. GEPA 反馈 Prompt

严格说，GEPA 这里不是完整的独立 Prompt 模板，而是系统构造的一段反馈文本。

内容包括：

- 当前关注字段
- `Expected output`
- `Actual output`
- `Observed error type`
- 为什么错
- 建议如何改 Prompt

来源：

- [backend/services/metric_factory.py](D:\project\Prompt-Optimization-Studio\backend\services\metric_factory.py)

它的作用是：

- 给 GEPA 这类 feedback-rich optimizer 提供错误反馈
- 帮优化器知道应该往哪个方向修 Prompt

它可以看成“优化反馈 Prompt”。

## 9. 报告摘要生成 Prompt

## 9.1 具体内容

当系统要生成 evaluation / optimization 的 executive summary 时，会构造一条摘要 Prompt：

- 为某类 report 写 concise executive summary
- 提供 metrics summary
- 提供 warnings
- 提供 representative failures
- 要求写得简短、具体，并指出可靠性风险

来源：

- [backend/services/report_summary.py](D:\project\Prompt-Optimization-Studio\backend\services\report_summary.py)

## 9.2 它的作用

这条 Prompt 不参与评测，也不参与优化。

它只负责把结构化运行结果转换成一段适合人读的摘要文字，用在报告页里。

可以把它理解为“报告解说 Prompt”。

## 9.3 回退机制

如果没有可用 LLM，系统不会硬失败，而是回退为模板化摘要。

这说明 executive summary 是增强体验，不是运行闭环的硬依赖。

## 10. OpenAI 结构化输出约束 Prompt

## 10.1 具体内容

在多个场景里，系统通过 `openai_client.generate_structured()` 调用结构化输出。

这时内部会自动加一段统一的 schema 约束提示，大意是：

- 只返回 JSON
- 不要 markdown fence
- 不要解释文字
- 返回结果必须严格匹配指定 schema
- 字段名必须完全一致

来源：

- [backend/services/openai_client.py](D:\project\Prompt-Optimization-Studio\backend\services\openai_client.py)

## 10.2 它会影响哪些地方

至少会影响这些 LLM 场景：

1. 合成数据生成
2. 自定义任务模板草稿生成
3. LLM Judge 的结构化判决

## 10.3 它的作用

这不是业务 Prompt，而是平台级“输出护栏 Prompt”。

它的作用是尽量把模型输出限制在可解析、可校验、可入库的范围内。

## 11. 优化完成后自动附加的 Prompt 文本

## 11.1 具体内容

优化完成后，系统不会只给出分数。

它会构造一个新的 `optimized_prompt`，并且在 `system_prompt` 后面追加类似这样的文字：

- `Optimizer note: refined by {optimizer_name}.`
- 如果 DSPy 学到了额外 instruction，还会加上：
  - `Learned instructions: ...`

来源：

- [backend/services/optimization_service.py](D:\project\Prompt-Optimization-Studio\backend\services\optimization_service.py)

## 11.2 它的作用

这段文字的作用是：

- 标记这条 Prompt 是优化器产物
- 把 DSPy 学到的指令显式暴露出来
- 让用户能在报告页直接看到优化后的 Prompt

## 11.3 它的意义

这让优化结果从“模型内部状态”变成“可读、可复制、可回写”的 Prompt 资产。

## 12. 当前项目里最关键的 Prompt 是哪几个

如果按业务价值排序，当前项目里最重要的 Prompt 是下面几类：

### 12.1 第一重要：业务 Prompt

也就是用户编辑的：

- `system_prompt`
- `user_template`

这是优化对象本身。

### 12.2 第二重要：DSPy 包装 Prompt

这是业务 Prompt 真正进入模型运行时的形态。

它决定：

- 输出会不会严格按 schema
- 模型是否只返回原始 JSON
- 业务 Prompt 是否被平台约束正确包裹

### 12.3 第三重要：评测与优化辅助 Prompt

包括：

- `llm_judge`
- `gepa_feedback_metric`
- executive summary

它们不直接产生业务预测，但会影响：

- 评分方式
- 优化反馈质量
- 报告可读性

## 13. 当前项目里“没有”的 Prompt

当前代码里也有一些值得明确说“还没有”的部分：

1. 没有预置的业务 Prompt 样本库
2. 没有按任务类型内置多套 Prompt 模板集合
3. 没有项目创建后自动生成并保存 baseline Prompt 记录
4. 没有把优化结果自动回写成新的 Prompt 版本

也就是说，当前系统已经具备 Prompt 生命周期能力，但还没有附带“Prompt 资产库”。

## 14. 一句话总结每类 Prompt 的用途

为了方便快速查阅，可以把当前项目中的提示词总结为：

- 业务 Prompt：真正要跑、要测、要优化的任务指令
- DSPy 包装 Prompt：把业务 Prompt 和 schema 拼成可执行指令
- 数据生成 Prompt：生成 synthetic dataset
- 模板草稿 Prompt：生成 custom task contract
- LLM Judge Prompt：把模型输出交给另一个模型评分
- GEPA 反馈 Prompt：给优化器提供可学习的错误反馈
- 报告摘要 Prompt：把运行结果翻译成可读摘要
- 结构化输出约束 Prompt：保证所有 LLM 返回 JSON 而不是散文
- 优化后附加 Prompt：把优化器学到的变化写回最终 Prompt

## 15. 总结

当前项目中的“提示词”并不只有用户在 Prompt Studio 页面看到的那两段文本。

从代码实现看，它实际上存在一个多层 Prompt 体系：

1. 最外层是用户编写的业务 Prompt
2. 中间层是平台为运行、结构化输出、评测和优化构造的系统 Prompt
3. 最内层是报告、反馈、配置生成这类辅助 Prompt

其中最核心的是业务 Prompt 和 DSPy 包装 Prompt，因为它们直接决定模型怎么做任务、怎么输出结果；而其余提示词则承担“造数据、评分、生成配置、生成摘要、提供优化反馈”的辅助作用。
