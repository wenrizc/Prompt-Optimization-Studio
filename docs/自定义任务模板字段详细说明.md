# 自定义任务模板字段详细说明

本文档用于解释下面这份自定义任务模板 JSON 中每个字段的含义、作用范围、典型写法，以及字段之间需要保持的一致性。

## 1. 示例配置

```json
{
  "task_key": "custom_task",
  "task_display_name": "Custom Task",
  "task_description": null,
  "input_schema_json": {
    "type": "object",
    "properties": {
      "text": {
        "type": "string"
      }
    },
    "required": [
      "text"
    ]
  },
  "output_schema_json": {
    "type": "object",
    "properties": {
      "answer": {
        "type": "string"
      }
    },
    "required": [
      "answer"
    ]
  },
  "default_metric_config_json": {
    "metric": "f1_token",
    "field": "answer",
    "correct_threshold": 0.8
  },
  "task_definition_json": {
    "task_family": "custom",
    "target_field": "answer"
  },
  "report_profile_json": {
    "task_family": "custom",
    "primary_output_field": "answer",
    "focus_areas": [
      "correctness"
    ]
  }
}
```

## 2. 这份配置整体是做什么的

这份 JSON 不是单纯的“展示信息”，而是一份任务契约。它会决定：

- 这个任务在系统里叫什么
- 样本输入长什么样
- 模型输出必须长什么样
- 默认用什么指标评测
- 系统把哪个输出字段当成主要目标
- 报告页优先展示哪个输出字段和哪些关注点

你可以把它理解成“一个项目模板的核心配置骨架”。

## 3. 顶层字段说明

### `task_key`

示例值：

```json
"task_key": "custom_task"
```

含义：

- 任务的系统内部唯一标识。
- 后端用它区分不同任务模板。
- 它更偏“程序标识”，而不是给最终用户看的名称。

作用范围：

- 模板保存到 `custom_task_templates` 表时作为唯一键。
- 项目创建后，项目记录里也会保存这个值。
- 代码中的很多逻辑会通过它识别任务类型来源。

命名要求：

- 必须以小写字母开头。
- 只能包含小写字母、数字、下划线。
- 长度需要符合后端校验规则。
- 不能使用保留关键字，例如 `builtin`、`custom`、`all`，也不能和内置任务键冲突。

建议：

- 用能表达任务语义的名字，例如 `sentiment_classification`、`faq_answering`、`contract_extraction`。
- 不建议使用过于泛的名字，如 `task1`、`demo_task`。

### `task_display_name`

示例值：

```json
"task_display_name": "Custom Task"
```

含义：

- 给用户看的任务显示名称。
- 更适合用于前端页面、下拉框、卡片标题、说明区域。

作用范围：

- 自定义任务模板列表展示。
- 项目详情和任务配置展示。
- 前端工作台中的可读名称。

建议：

- 这个字段应当清楚表达业务含义。
- 如果面向中文用户，建议直接写中文，例如 `历史事实问答`、`合同字段抽取`。

### `task_description`

示例值：

```json
"task_description": null
```

含义：

- 对任务目标和处理规则的补充说明。
- 可以为空，但建议填写。

作用范围：

- 前端配置展示。
- 数据生成时，LLM 会参考这里的描述。
- 后续做项目理解、报告解释或提示词设计时，这个字段能提供额外上下文。

适合写的内容：

- 任务要解决什么问题
- 输入通常是什么
- 输出应该包含什么
- 有哪些关键约束

示例：

```json
"task_description": "根据用户输入的问题，返回简洁、准确的历史事实答案。"
```

## 4. 输入输出契约字段

### `input_schema_json`

示例值：

```json
{
  "type": "object",
  "properties": {
    "text": {
      "type": "string"
    }
  },
  "required": [
    "text"
  ]
}
```

含义：

- 定义单条样本输入的 JSON 结构。
- 它告诉系统“每条输入数据应该长什么样”。

当前这份配置表示：

- 输入是一个对象
- 对象里有一个 `text` 字段
- `text` 是字符串
- `text` 是必填字段

作用范围：

- 数据集样本的 `input_json` 应该和这里匹配。
- Prompt 渲染通常会使用其中的字段，当前项目尤其依赖 `text`。
- 合成生成数据集时，系统会把它发给 LLM 作为输入结构说明。

为什么这里通常写 `text`：

- 当前项目的 Prompt 模板校验默认要求 `user_template` 包含 `{text}`。
- DSPy 运行桥接层也主要围绕 `text -> answer` 这条主路径工作。

建议：

- 如果你想走当前项目最稳的路径，输入 schema 里最好保留 `text` 字段。
- 如果以后扩展成多字段输入，也建议先确认前后端和运行链路都支持。

### `output_schema_json`

示例值：

```json
{
  "type": "object",
  "properties": {
    "answer": {
      "type": "string"
    }
  },
  "required": [
    "answer"
  ]
}
```

含义：

- 定义模型输出的标准结构。
- 它告诉系统“模型最终必须产出什么格式的数据”。

当前这份配置表示：

- 输出是一个对象
- 对象里有一个 `answer` 字段
- `answer` 是字符串
- `answer` 是必填字段

作用范围：

- 评测时会用它来校验预测输出结构。
- 合成数据生成时会要求 LLM 的 `expected_output` 严格匹配它。
- DSPy 运行时也会把它拼进约束指令里，要求模型只返回符合 schema 的 JSON。

建议：

- 当前项目最稳定的形态是顶层 `object`。
- 字段名要尽量简洁明确，例如 `answer`、`label`、`score`、`category`。
- 如果后续要做指标计算、报告展示或优化，关键字段命名要前后一致。

## 5. 默认评测配置字段

### `default_metric_config_json`

示例值：

```json
{
  "metric": "f1_token",
  "field": "answer",
  "correct_threshold": 0.8
}
```

含义：

- 定义这个任务默认使用什么评测指标。
- 项目创建后，evaluation 和 optimization 会默认复用这里的配置。

字段说明：

- `metric`
  - 指标名称。
  - 当前示例是 `f1_token`，表示做分词级别的 F1 匹配。
- `field`
  - 评测时主要取输出中的哪个字段来算分。
  - 当前写的是 `answer`，表示会拿 `output.answer` 做对比。
- `correct_threshold`
  - 将连续分数视为“正确”的阈值。
  - 例如 `0.8` 表示分数达到 0.8 及以上才算通过。

为什么这里写 `answer`：

- 因为 `output_schema_json.properties` 里定义的关键输出字段就是 `answer`。
- 后端也要求这个字段必须真实存在于输出 schema 中。

适用场景说明：

- `f1_token` 适合文本答案允许部分重合、不是必须完全一样的任务。
- 如果是分类任务，很多时候 `exact_match` 或 `normalized_exact_match` 更直观。
- 如果是结构化多字段输出，可以考虑 JSON 字段级指标。

## 6. 任务定义字段

### `task_definition_json`

示例值：

```json
{
  "task_family": "custom",
  "target_field": "answer"
}
```

含义：

- 这是系统对“这个任务本质上在优化什么”的抽象定义。
- 它更偏平台内部配置，不只是展示文案。

字段说明：

- `task_family`
  - 任务家族或任务类别。
  - 当前示例用 `custom`，表示这是一个用户自定义任务。
- `target_field`
  - 任务最核心的目标输出字段。
  - 当前写的是 `answer`，表示系统把 `answer` 视为主要优化对象。

作用范围：

- 自定义模板生成与校验时会检查它是否和输出 schema 对齐。
- 运行链路在理解“主要输出字段”时会参考这里。

建议：

- `target_field` 必须和 `output_schema_json.properties` 中的字段名一致。
- 如果输出有多个字段，`target_field` 应该指向最重要、最具业务代表性的那个。

## 7. 报告展示配置字段

### `report_profile_json`

示例值：

```json
{
  "task_family": "custom",
  "primary_output_field": "answer",
  "focus_areas": [
    "correctness"
  ]
}
```

含义：

- 定义报告页应该重点强调什么。
- 它更偏“结果展示视角”，而不是运行执行本身。

字段说明：

- `task_family`
  - 报告展示所对应的任务家族。
  - 这里通常和 `task_definition_json.task_family` 保持一致。
- `primary_output_field`
  - 报告页最主要展示的输出字段。
  - 当前为 `answer`。
- `focus_areas`
  - 报告关注点列表。
  - 当前只有 `correctness`，表示报告重点关注正确性。

`focus_areas` 可以怎么理解：

- `correctness`：关注答对没答对
- `completeness`：关注答得是否完整
- `consistency`：关注输出是否稳定一致
- `format_compliance`：关注格式是否符合 schema

注意：

- 后端要求 `focus_areas` 必须是非空字符串数组。
- `primary_output_field` 也必须真实存在于 `output_schema_json.properties` 中。

## 8. 这几个字段之间必须保持什么一致性

这份配置最关键的不是“每个字段单独合法”，而是“彼此要对齐”。

当前这份示例中的一致性关系是：

- `output_schema_json.properties` 里定义了 `answer`
- `default_metric_config_json.field` 也写的是 `answer`
- `task_definition_json.target_field` 也写的是 `answer`
- `report_profile_json.primary_output_field` 也写的是 `answer`

也就是说，系统从评测、任务定义、报告展示三个角度，都认定 `answer` 是核心输出字段。

如果这几个地方不一致，会出现的问题通常包括：

- 指标算分字段找不到
- 模板生成校验失败
- 报告无法正确聚焦主字段
- 后续优化逻辑难以判断主要目标

## 9. 针对这份示例配置的通俗理解

这份配置可以翻译成一句更容易理解的话：

“这是一个自定义问答任务。每条输入只有一个 `text` 文本字段，模型需要输出一个字符串类型的 `answer` 字段。系统默认用 `f1_token` 去衡量这个 `answer` 和标准答案有多接近，分数达到 0.8 以上就认为回答正确。后续报告和任务定义也都把 `answer` 当成最核心的输出字段。”

## 10. 这份配置适合什么任务

这份配置尤其适合下面这类任务：

- 简单问答
- 短文本回答
- FAQ 回答
- 历史事实类回答
- 允许部分词语差异、但关注核心语义接近的回答任务

如果你的任务是下面这些类型，通常就需要调整：

- 分类任务
  - 可以把输出字段改成 `label`
  - metric 可能更适合 `exact_match`
- 评分任务
  - 可以把输出字段改成 `score`
  - schema 可能要用数值类型
- 多字段抽取任务
  - 输出 schema 需要定义多个字段
  - metric 和 `primary_output_field` 也要相应调整

## 12. 最后给一个实操建议

如果你在创建自定义任务模板时不确定怎么配，优先检查这 4 个地方是不是统一指向同一个主输出字段：

- `output_schema_json.properties`
- `default_metric_config_json.field`
- `task_definition_json.target_field`
- `report_profile_json.primary_output_field`

只要这 4 处是统一的，再加上 `input_schema_json` 和 `output_schema_json` 都是清晰的顶层对象结构，这个模板通常就能比较稳定地跑通后续的数据生成、评测、优化和报告展示。
