# Prompt Optimization Studio 项目架构与功能设计详解

## 1. 项目定位

`Prompt Optimization Studio` 是一个面向本地运行场景的 Prompt 优化工作台，核心目标是把 Prompt 工程中的几个关键环节串成闭环：

1. 定义任务
2. 创建项目
3. 准备数据集
4. 编写和版本化 Prompt
5. 执行评测
6. 执行 Prompt 优化
7. 查看报告、制品和运行日志

从代码实现看，它不是一个“纯算法库”，而是一个完整的本地应用：

- 后端：FastAPI + SQLAlchemy + SQLite
- 执行层：本地 Worker + 数据库任务队列
- 模型层：DSPy + OpenAI/OpenAI-compatible 接口
- 前端：Next.js 15 + React 19
- 存储层：数据库保存结构化元数据，磁盘保存报告与制品文件

它的设计重点不是高并发分布式调度，而是“本地可跑通、结构清晰、可逐步扩展”。

## 2. 总体架构

系统可以分成 5 层：

### 2.1 展示层

位于 `frontend/`，提供项目、数据集、Prompt、运行、报告、自定义任务等页面。

这一层的职责是：

- 调用后端 REST API
- 组织工作流页面
- 持久化当前选择的项目/数据集/Prompt
- 展示结构化 JSON、状态、报告和制品

### 2.2 API 层

位于 `backend/api/routes/`，负责：

- 参数接收与响应输出
- 路由聚合
- 数据库依赖注入
- 调用服务层完成业务逻辑

API 层尽量保持薄，只做资源编排，不承载复杂业务。

### 2.3 业务服务层

位于 `backend/services/`，是本项目的核心。这里实现了：

- 数据集导入、生成、拆分与质量摘要
- Prompt 校验与版本化
- Evaluation 创建、执行、报告生成
- Optimization Run 创建、执行、报告生成
- 指标计算
- DSPy 程序构造
- Artifact 制品落盘与 manifest 维护
- OpenAI 结构化调用封装

### 2.4 执行与调度层

位于 `backend/workers/` 与 `backend/services/job_service.py`。

这一层把耗时任务抽象成 Job：

- Evaluation 会创建一个后台 Job
- Optimization Run 也会创建一个后台 Job
- Worker 从数据库中 claim Job
- 按任务类型路由到对应执行函数
- 持续更新进度、状态和日志

这本质上是一个“轻量本地任务队列”。

### 2.5 持久化层

分为两部分：

- SQLite：保存项目、Prompt、数据集、样本、评测、优化运行、任务、日志、制品索引
- 文件系统：保存 report、predictions、optimized_prompt、compiled_program 等 JSON 文件

因此系统采用了“数据库索引 + 文件制品正文”的组合存储模型。

## 3. 目录结构与模块职责

### 3.1 后端目录

`backend/`

- `main.py`：FastAPI 应用工厂，注册路由、CORS、生命周期、静态文件挂载
- `cli.py`：命令行入口，启动 Uvicorn
- `core/`：配置、常量、异常、运行时目录
- `db/`：SQLAlchemy 基类、engine、session
- `models/`：ORM 模型
- `schemas/`：Pydantic 请求/响应模型
- `api/routes/`：REST API
- `services/`：核心业务逻辑
- `workers/`：后台任务领取与执行

### 3.2 前端目录

`frontend/`

- `app/`：Next.js App Router 页面
- `components/`：通用 UI 组件
- `lib/api.ts`：统一请求封装
- `lib/studio.ts`：前端类型定义和工具函数
- `lib/use-studio-workspace.ts`：工作空间状态管理
- `lib/i18n/`：多语言支持

### 3.3 测试目录

`tests/`

- 当前只有 CLI 入口相关测试，验证命令行脚本配置与启动路径

说明当前测试覆盖偏薄，主要验证“能启动”，而非核心业务逻辑。

## 4. 后端核心设计

## 4.1 应用启动与生命周期

入口在 `backend/main.py`。

启动时会做三件事：

1. `ensure_runtime_directories()`：确保 `data/` 相关目录存在
2. `Base.metadata.create_all(bind=engine)`：按 ORM 模型自动建表
3. 如果存在 `frontend/out/`，将其挂载为静态站点根目录

这意味着：

- 后端 API 和前端静态页面可以由同一个 FastAPI 进程提供
- 本地部署非常轻量
- 不强依赖额外静态资源服务

## 4.2 配置系统

配置定义在 `backend/core/config.py`，使用 `pydantic-settings` 从 `.env` 加载。

关键配置包括：

- 服务地址：`host`、`port`
- 数据目录：`data_dir`
- 数据库：`database_url`
- LLM：`openai_api_key`、`openai_base_url`、`openai_default_model`
- 资源限制：最大上传、最大 artifact、最大运行样本数等

同时通过 `computed_field` 动态导出：

- `uploads_dir`
- `artifacts_dir`
- `reports_dir`
- `generated_dir`

这种设计把“路径约定”集中管理，避免业务代码散落硬编码目录。

## 4.3 数据库模型设计

### 4.3.1 Project

`Project` 是顶层工作空间实体，描述“任务是什么”。

它不仅保存名称和描述，还保存任务契约：

- `task_kind`
- `task_key`
- `task_display_name`
- `task_description`
- `input_schema_json`
- `output_schema_json`
- `default_metric_config_json`
- `task_definition_json`
- `report_profile_json`

这说明项目不是简单分组容器，而是整个评测/优化流程的“任务配置中心”。

### 4.3.2 Prompt

`Prompt` 归属于某个项目，并通过 `(project_id, name, version)` 唯一约束实现版本管理。

关键字段：

- `system_prompt`
- `user_template`
- `output_schema_json`
- `version`

系统允许同名 Prompt 多版本演进，这是后续评测和优化的重要基础。

### 4.3.3 Dataset 与 DatasetExample

`Dataset` 保存元信息：

- 来源类型
- schema
- 生成指令
- 生成模型
- 质量摘要

`DatasetExample` 保存具体样本：

- `input_json`
- `expected_output_json`
- `metadata_json`
- `split`
- `content_hash`

这里的关键设计点是：

- 样本输入和期望输出都使用 JSON，而非固定字段
- 配合项目级 schema，可以支持多种任务类型
- `content_hash` 为重复检测和数据质量分析提供基础

### 4.3.4 Evaluation

`Evaluation` 表示某个 Prompt 在某个数据集上的一次评测。

它会冻结运行时快照：

- `metric_config_json`
- `model_config_json`
- `prompt_snapshot_json`
- `dataset_split_snapshot_json`
- `package_versions_json`

这是一种很重要的设计：运行时依赖会被“冻结”，避免后续项目、Prompt 或数据集变更导致结果不可复现。

### 4.3.5 OptimizationRun

`OptimizationRun` 表示一次 Prompt 优化任务。

除基础关联外，还会保存：

- `optimizer_name`
- `baseline_score`
- `optimized_score`
- `optimizer_config_snapshot_json`
- `metric_config_snapshot_json`
- `model_config_snapshot_json`

和 Evaluation 一样，它也是“快照优先”的设计。

### 4.3.6 Job

`Job` 是后台任务队列核心。

关键字段：

- `job_type`
- `target_type`
- `target_id`
- `status`
- `progress`
- `locked_by`
- `locked_until`
- `heartbeat_at`
- `idempotency_key`

这使它具备最小可用的任务队列能力：

- 排队
- 领取
- 乐观锁防重入
- 进度更新
- 取消

### 4.3.7 Artifact

`Artifact` 不直接保存正文，而是保存文件索引：

- `owner_type`
- `owner_id`
- `artifact_type`
- `relative_path`
- `metadata_json`

正文在磁盘 JSON 文件里。

### 4.3.8 RunLog

`RunLog` 记录运行中产生的过程日志，用于：

- 任务进度追踪
- 报错排查
- 前端展示运行历史

## 4.4 API 设计

路由由 `backend/api/router.py` 聚合，主要资源包括：

- `/projects`
- `/custom-task-templates`
- `/prompts`
- `/datasets`
- `/evaluations`
- `/optimization-runs`
- `/worker`
- `/jobs`
- `/artifacts`
- `/run-logs`

这些路由基本遵循“资源式设计”，并围绕工作流拆分。

### 4.4.1 Projects

支持：

- 内置任务目录查询
- 项目创建
- 项目列表
- 项目详情
- 项目更新
- 项目归档

项目创建时会根据 `task_kind` 和 `task_key` 自动补足模板字段。

### 4.4.2 Prompts

支持：

- Prompt 模板校验
- 创建 Prompt
- 查询 Prompt
- 基于既有 Prompt 创建新版本

这里体现的是“版本追加”而不是原地修改。

### 4.4.3 Datasets

支持：

- 创建空数据集
- 导入数据集
- 生成合成数据集
- 查看样本
- 修改样本
- 自动拆分 train/dev/test
- 生成质量报告

### 4.4.4 Evaluations 与 Optimization Runs

支持：

- 创建运行任务
- 列表查询
- 详情查询
- 报告读取
- 取消任务

Evaluation 与 Optimization 都不是同步执行，而是“创建记录 + 创建 Job”。

### 4.4.5 Worker / Jobs / Logs / Artifacts

这是运行可观测性的基础设施接口：

- Worker 手动跑一次
- Job 列表与取消
- RunLog 查询
- Artifact 列表、manifest、内容预览、文件下载

## 5. 任务模板体系

项目的一个核心设计是：先定义任务，再驱动后续全部流程。

## 5.1 内置任务模板

定义在 `backend/services/task_catalog.py`。

当前代码中的内置任务有 3 个：

- `qa`
- `json_generation`
- `rate`

每个模板同时定义：

- 展示名
- 描述
- 输入 schema
- 输出 schema
- 默认 metric
- task_definition
- report_profile

这使得“项目创建”可以半自动完成，不需要用户每次从零配置全部字段。

## 5.2 自定义任务模板

自定义模板由 `custom_task_templates` 资源管理。

支持两种方式：

1. 手工编辑 JSON 配置
2. 用自然语言调用 `/generate-draft` 生成草稿

草稿生成服务 `custom_task_template_generator.py` 会调用结构化 LLM 输出，并要求：

- `task_key`
- schema
- metric 字段
- `target_field`
- `primary_output_field`

这些字段内部必须对齐。

这说明系统不只是“让模型随便生成一段模板”，而是明确要求模板配置之间保持契约一致。

## 6. 数据集能力设计

## 6.1 导入

`dataset_service.py` 支持：

- JSON
- JSONL
- CSV

导入时会：

1. 解析源内容
2. 归一化输入输出结构
3. 计算样本 `content_hash`
4. 写入数据库
5. 刷新质量摘要
6. 保存导入快照

导入快照是很实用的设计，它保留了原始输入，便于追溯。

## 6.2 合成生成

`dataset_generation.py` 支持两种模式：

- `mock`
- `openai`

当未配置 `OPENAI_API_KEY` 时，系统回退为 mock，生成可用但简单的样例。

配置 LLM 后，则：

1. 根据项目 `output_schema_json` 动态构造 Pydantic 模型
2. 调用结构化生成接口
3. 要求返回的 JSON 严格满足 schema
4. 导入为正式数据集样本

这意味着生成数据集并不是自由文本，而是“结构化合成”。

## 6.3 数据集拆分

`apply_dataset_split()` 会按比例把样本分配到：

- train
- dev
- test

并支持按字段分层。

由于优化器对数据划分敏感，拆分是运行前的关键步骤。

## 6.4 质量摘要

`refresh_dataset_quality_summary()` 会生成质量摘要，供运行前校验和报告展示使用。

质量摘要会服务于几个目的：

- 看 split 分布是否合理
- 看标签分布
- 检查跨 split 重复
- 识别 synthetic / real 样本比例

特别是跨 split 重复检测，会直接阻止评测执行，防止数据泄漏。

## 7. Prompt 设计

Prompt 在系统里由两部分组成：

- `system_prompt`
- `user_template`

其中 `user_template` 支持变量替换，比如 `{text}`。

在 `dspy_program_factory.py` 中，系统会把 Prompt 转成 DSPy 指令：

1. 拼接系统提示
2. 拼接用户模板
3. 附加输出 schema
4. 强制要求只返回符合 schema 的 JSON

这说明 Prompt 在运行时并不是裸文本透传，而是被包装成一个更完整的“任务执行指令”。

### 7.1 Prompt 校验

`validators.py` 中负责：

- 模板变量提取
- 输出 schema 兼容性检查

前端也提供了单独的“Validate”操作，先校验后保存。

### 7.2 Prompt 版本化

新版本不是覆盖旧记录，而是插入 `version + 1`。

这样做有两个好处：

1. 评测和优化永远可以引用历史版本
2. 结果可追溯，不会被后续编辑污染

## 8. Evaluation 评测实现原理

## 8.1 创建阶段

`create_evaluation_and_job()` 会：

1. 校验项目、数据集、Prompt 归属关系
2. 解析项目默认 metric
3. 解析默认模型配置
4. 冻结 Prompt 快照
5. 冻结数据集 split 快照
6. 校验输入合法性
7. 创建 `Evaluation`
8. 创建关联 `Job`
9. 记录运行日志

核心思想是：提交任务时就固化所有依赖。

## 8.2 执行阶段

Worker 路由到 `execute_evaluation()` 后，会：

1. 读取数据集和样本
2. 刷新质量摘要
3. 检查样本输出是否符合 schema
4. 构建 DSPy 程序
5. 配置运行时
6. 对每个样本生成预测
7. 调用 metric 计算分数
8. 聚合 overall score
9. 生成报告

### 8.2.1 预测生成

`generate_prediction()` 中：

- `mock` 模式直接返回 `expected_output_json`
- 真实模式调用 DSPy 程序，拿到 `answer`
- 再按 schema 解析成结构化 JSON

这保证了本地无模型时也能跑通链路。

### 8.2.2 指标计算

`metric_factory.py` 支持：

- `exact_match`
- `normalized_exact_match`
- `contains`
- `f1_token`
- `json_field_accuracy`
- `json_all_fields_accuracy`
- `weighted_numeric_fields_accuracy`
- `llm_judge`
- `gepa_feedback_metric`

也就是说，本项目不仅支持简单字符串匹配，还支持：

- JSON 字段级评分
- 数值型多字段加权评分
- LLM Judge
- 面向 GEPA 的带反馈指标

### 8.2.3 报告生成

`build_evaluation_report()` 会输出：

- summary
- executive_summary
- dataset 概要
- metric 配置
- score_breakdown
- failed_examples
- warnings
- results
- artifacts
- program_state

这里的 `executive_summary` 来自 `report_summary.py`，用于生成更适合阅读的高层总结。

## 8.3 成功收尾

`finalize_evaluation_success()` 会：

- 更新状态和分数
- 写 `report.json`
- 写 `predictions.json`
- 记录运行日志

这说明“数据库里存摘要，文件里存详细制品”的设计贯彻得很一致。

## 9. Optimization Run 优化实现原理

Optimization 的整体结构和 Evaluation 类似，但流程更长。

## 9.1 创建阶段

`create_optimization_run_and_job()` 会：

1. 校验项目、数据集、Prompt
2. 冻结 Prompt/数据集/model/metric/optimizer 快照
3. 校验优化器前置条件
4. 创建 `OptimizationRun`
5. 创建 Job
6. 写运行日志

## 9.2 优化器前置校验

`validate_optimizer_inputs()` 根据优化器类型检查：

- `bootstrap_fewshot`：必须有 train
- `miprov2`：必须有 dev，且总样本数至少 30
- `gepa`：必须有 dev，必须使用 feedback-rich metric，且总样本数至少 30

这体现了一个很好的工程原则：尽量在执行前失败，而不是跑到一半才发现条件不满足。

## 9.3 执行流程

`execute_optimization_run()` 主要包括：

1. 加载数据集和样本
2. 构建 baseline DSPy program
3. 把样本转成 train/dev/test bundle
4. 根据优化器名称编译 optimized program
5. 从 optimized program 提取新的 Prompt 信息
6. 对 baseline 和 optimized 分别跑预测
7. 计算 baseline/optimized score
8. 构建对比报告
9. 导出编译程序状态和 few-shot demos

### 9.3.1 DSPy 编译

`compile_optimized_program()` 根据 `optimizer_name` 分发到：

- `_compile_bootstrap_fewshot`
- `_compile_miprov2`
- `_compile_gepa`

它们都是对 DSPy 官方优化器的薄封装，重点在于：

- 接受项目内统一的 metric
- 接受系统已有的数据 split
- 接受 JSON 化的优化器配置

### 9.3.2 优化后的 Prompt

`build_optimized_prompt()` 的做法很直接：

- 保留原 `user_template`
- 在 `system_prompt` 末尾拼接 optimizer note
- 如果 DSPy 学到了指令，则附加 learned instructions

也就是说，这里优化结果不是“神秘黑盒”，而是转回可查看的 Prompt 表达。

### 9.3.3 对比报告

优化报告会输出：

- baseline / optimized / delta
- split 维度分数
- error type 维度分数
- failed examples
- regression examples
- optimized_prompt
- prompt_diff
- compiled_program
- fewshot_demos

这个设计对调试很有价值，因为不仅告诉你“分数有没有提升”，还告诉你“Prompt 被改成什么了”。

## 10. DSPy 程序构建原理

`dspy_program_factory.py` 是本项目非常关键的桥接层。

它做了两件事：

### 10.1 Prompt 到 DSPy Module 的转换

`PromptOptimizationProgram` 是一个简单的 `dspy.Module`：

- Signature 固定为 `text -> answer`
- 使用组装后的 instructions 作为指令

这意味着系统把所有任务统一映射到“输入 text，输出 answer/JSON 文本”的形态，再由解析器还原结构化输出。

### 10.2 样本到 DSPy Example 的转换

每个 `DatasetExample` 会被转成 DSPy Example，并附带：

- `text`
- `answer`
- `expected_output_json`
- `example_id`
- `split`

训练、验证、测试集合也是在这里按 split 切分。

这层让数据集和 DSPy 之间解耦，后续若更换运行框架，只需要替换桥接层。

## 11. 模型调用与运行时适配

## 11.1 运行模式

运行模式由 `run_defaults.py` 和 `dspy_runtime.py` 决定：

- 未配置 `OPENAI_API_KEY`：`provider=mock`
- 配置后：`provider=openai`

这是本项目本地可用性的关键设计。

## 11.2 OpenAI 与兼容接口适配

`openai_client.py` 兼容两类接口：

1. 官方 OpenAI `responses` API
2. OpenAI-compatible 的 `chat.completions`

判断逻辑基于 `OPENAI_BASE_URL`。

结构化生成时：

- 官方 OpenAI 走 `responses.parse`
- 兼容服务走 `chat.completions + response_format=json_object`
- 再由本地 Pydantic 校验

这让项目能兼容 DeepSeek 等网关。

## 11.3 DSPy Runtime

`configure_runtime()` 会：

- 在 mock 模式返回空运行时
- 在真实模式实例化 `dspy.LM`
- 根据 base_url 推断 `responses` 或 `chat` 模式
- 对 DeepSeek 注入 `thinking.disabled`

说明作者已经把“兼容模型差异”的工程问题纳入考虑，而不是只适配单一官方 API。

## 12. 制品与报告体系

## 12.1 Artifact 存储模型

`artifact_service.py` 的设计很清晰：

1. JSON 序列化内容
2. 写入 `data/artifacts/...`
3. 计算 `sha256`
4. 更新数据库索引
5. 重写 `manifest.json`

manifest 的作用是把某次运行的全部制品组织成清单，便于前端统一展示。

## 12.2 安全设计

`resolve_artifact_path()` 和 manifest 读取都做了路径逃逸校验，确保访问不能跳出 `data_dir`。

这是一处比较扎实的本地文件访问安全处理。

## 12.3 报告结构

报告不是简单文本，而是结构化 JSON。

这样带来两个好处：

1. 前端可以按卡片、表格、JSON 预览自由渲染
2. 后续可以扩展更多自动分析字段，而不破坏现有结构

## 13. Job 与 Worker 调度设计

## 13.1 Job 创建

耗时任务在创建实体后，会立刻创建 Job。

Job 使用 `idempotency_key` 防止重复提交。

## 13.2 Job 领取

`claim_next_job()` 的实现是：

1. 查询最早 `queued` 的 job id
2. 用 `update ... where status='queued'` 原子更新
3. 若 `rowcount == 0`，说明已被其他 worker 抢走

这是一个典型的“乐观锁领取任务”实现，简单但有效。

## 13.3 Job 生命周期

Job 状态包含：

- `queued`
- `running`
- `cancel_requested`
- `cancelled`
- `succeeded`
- `failed`

配合：

- `heartbeat_at`
- `locked_until`
- `progress`

可以支持最小化的任务监控。

## 13.4 Worker

`LocalWorker.run_once()` 是串行轮询执行：

1. claim 一个 job
2. 按类型分发
3. 成功则 complete
4. 失败则 fail

当前不是持续驻守进程，而是通过 API 主动触发一次运行。这非常适合本地调试和演示。

## 14. 前端架构设计

## 14.1 路由与本地化

前端使用 Next.js App Router，并采用 `/{lang}/...` 路由结构。

根路径页面只负责根据浏览器语言重定向到：

- `/en/...`
- `/zh/...`

`I18nProvider` 提供：

- `t()` 文本翻译
- `tm()` 结构化消息读取
- `href()` 本地化链接拼接
- `switchLocalePath()` 语言切换

## 14.2 工作空间状态

`useStudioWorkspace()` 管理：

- 项目列表
- 数据集列表
- Prompt 列表
- 当前选中的 project/dataset/prompt

并把选择写入 `localStorage`。

这让各页面可以共享当前上下文，而不需要引入更重的全局状态管理。

## 15. 关键业务闭环

## 15.1 从项目到评测

完整链路如下：

1. 创建 Project，确定任务 schema 与默认 metric
2. 创建或导入 Dataset
3. 创建 Prompt
4. 创建 Evaluation
5. Evaluation 生成 Job
6. Worker 执行 Job
7. 生成 report 和 predictions artifact
8. 前端读取报告与日志

## 15.2 从评测到优化

完整链路如下：

1. 基于已有 Project + Dataset + Prompt 创建 OptimizationRun
2. 记录 optimizer 配置快照
3. Worker 运行 DSPy 优化器
4. 输出优化后 Prompt、对比结果、few-shot demos
5. 前端报告页展示优化结果与差异

## 16. 设计上的优点

### 16.1 快照化设计合理

Evaluation 和 OptimizationRun 都固化了 Prompt、Dataset split、metric、model、依赖版本。

这使得结果具备可追溯性和相对可复现性。

### 16.2 本地闭环完整

即使没有真实模型，也能通过 mock 路径跑通：

- 数据生成
- 评测
- 优化
- 报告

这非常适合开发和产品验证。

### 16.3 任务契约抽象清晰

项目以 schema、metric、task_definition 为核心，而不是把逻辑写死在代码里。

这为后续扩展更多任务类型打下了基础。

### 16.4 制品体系实用

不是只返回分数，而是保存：

- 报告
- 预测
- 编译状态
- 优化后 Prompt
- Prompt diff
- few-shot demos

这让系统更像一个真正可调试的工作台。

## 17. 当前实现的边界与注意点

基于当前代码，可以看出以下边界：

### 17.1 内置任务数量有限

当前常量与任务目录里只稳定实现了 `qa`、`json_generation`、`rate` 三类。

如果 README 中提到更多任务类型，应以当前代码为准。

### 17.2 Worker 是手动触发式

现在没有常驻异步任务消费进程，而是通过 `/api/v1/worker/run-once` 手动驱动。

这对本地演示足够，但不等于生产级调度系统。

### 17.3 测试覆盖较少

当前测试主要覆盖 CLI 入口，核心业务如数据拆分、评测、优化、artifact 管理尚未看到系统化测试。

### 17.4 前端偏工作台而非强约束表单

大量 JSON 配置直接暴露给用户编辑，灵活性高，但对使用者的 schema 和 metric 理解有一定要求。

## 18. 适合的后续演进方向

如果后续继续演进，这个架构比较适合扩展以下能力：

- 增加更多内置任务模板
- 为 Dataset / Evaluation / Optimization 增加更完整测试
- 引入常驻 worker 进程或定时调度
- 增加更多指标实现
- 支持更细粒度的 artifact 类型和报告渲染
- 支持项目级运行历史对比
- 增加 Prompt 优化结果回写为新版本 Prompt 的能力

## 19. 总结

这个项目的核心不是单点功能，而是把 Prompt 工程中的“任务定义、数据准备、Prompt 管理、评测、优化、报告”做成一个统一工作流。

从代码结构上看，它采用了非常典型且清晰的分层：

- 前端负责工作台体验
- API 层负责资源暴露
- 服务层承载核心业务
- Job/Worker 负责耗时任务
- 数据库 + 文件系统共同完成持久化

从实现原理上看，最重要的几个设计抓手是：

- 任务契约中心化
- 运行快照冻结
- 本地可运行的 mock/openai 双模式
- DSPy 作为优化执行引擎
- 报告与 artifact 的结构化保存

因此，它本质上是一个“本地 Prompt 优化实验平台”，已经具备较完整的工程骨架，并且扩展方向也比较明确。
