# Prompt Optimization Studio 完整设计方案

## 1. 产品定位

Prompt Optimization Studio 是一个本地单机 Web 应用，用于帮助用户完成从任务定义、数据集构建、Prompt 编写、自动优化到评测报告的完整流程。

系统目标不是训练模型，而是优化：

* system prompt；
* user prompt template；
* task instructions；
* few-shot examples；
* DSPy compiled program；
* 输出格式稳定性；
* 任务评测流程；
* 失败样本分析。

核心用户路径：

```text
创建项目
  ↓
选择内置任务或创建自定义任务
  ↓
定义输入 / 输出 schema
  ↓
导入或生成数据集
  ↓
审核和拆分样例
  ↓
创建 baseline prompt
  ↓
运行 baseline evaluation
  ↓
选择 BootstrapFewShot / MIPROv2 / GEPA
  ↓
执行优化
  ↓
查看报告、失败样本、prompt diff、few-shot demos
  ↓
导出 optimized prompt / compiled artifact / report
```

## 2.1 本阶段支持

| 模块                       | 是否支持                          |
| ------------------------ | ----------------------------- |
| 项目管理                     | 支持                            |
| 内置任务类型                   | 支持                            |
| 用户自定义英文任务名               | 支持                            |
| 输入 / 输出 schema 定义        | 支持                            |

| Baseline evaluation      | 支持                            |
| BootstrapFewShot         | 支持                            |
| MIPROv2                  | 支持                            |
| GEPA                     | 支持，但必须使用 feedback-rich metric |

## 2.2 本阶段不支持

| 不支持项              | 原因            |
| ----------------- | ------------- |
| Fine-tuning       | 本阶段已明确移除      |

| RAG pipeline 编排   | 需要额外检索和索引系统   |
| 多用户权限系统           | 本地单机 MVP 暂不需要 |

| 用户自定义 DSPy Module | 安全和复杂度过高      |
| 云端部署              | 当前目标是本地单机版    |

# 3. 任务类型设计

系统采用：

```text
task_kind = builtin | custom
task_key = 内置任务名或用户自定义英文任务名
```

不再使用 `other`。用户自定义任务直接使用英文 `task_key` 表示。

---

## 3.1 内置任务

| task_kind | task_key        | 说明               |
| --------- | --------------- | ---------------- |
| builtin   | classification  | 文本分类，例如意图分类、风险分类 |
| builtin   | extraction      | 信息抽取，例如从文本中抽取字段  |
| builtin   | qa              | 问答任务             |
| builtin   | json_generation | 根据输入生成结构化 JSON   |
| builtin   | rewriting       | 改写、润色、风格转换       |
| builtin   | rate            | 对一段话进行多维度打分      |

---

## 3.2 自定义任务

用户可以创建自定义英文任务名。

示例：

| task_kind | task_key              | 展示名      |
| --------- | --------------------- | -------- |
| custom    | ad_compliance_review  | 广告合规审核   |
| custom    | contract_risk_check   | 合同风险检查   |
| custom    | essay_feedback        | 作文反馈     |
| custom    | support_reply_quality | 客服回复质量评估 |
| custom    | resume_screening      | 简历筛选     |

自定义 `task_key` 规则：

```text
只能包含小写英文字母、数字、下划线
必须以字母开头
长度建议 3–50
不能和内置 task_key 重名
不能使用系统保留词
```

合法示例：

```text
ad_compliance_review
essay_scoring
contract_risk_check
```

非法示例：

```text
广告审核
AdCompliance
123_task
classification
rate
```

# 4. 自定义任务设计

自定义任务不是让用户写代码，而是通过配置定义任务。

用户需要定义：

| 内容                         | 说明     |
| -------------------------- | ------ |
| task_key                   | 英文任务名  |
| task_display_name          | 展示名    |
| task_description           | 任务说明   |
| input_schema_json          | 输入结构   |
| output_schema_json         | 输出结构   |
| default_metric_config_json | 默认评测方式 |
| report_profile_json        | 报告展示方式 |
| task_definition_json       | 完整任务定义 |

示例：

```text
task_key: ad_compliance_review
task_display_name: 广告合规审核
task_description: 判断广告文案是否存在违规风险，并输出风险等级、违规点和解释。
```

输出结构：

| 字段          | 类型       | 说明                  |
| ----------- | -------- | ------------------- |
| risk_level  | enum     | low / medium / high |
| violations  | string[] | 违规点                 |
| explanation | string   | 判断理由                |

默认 metric：

| 字段          | metric              |
| ----------- | ------------------- |
| risk_level  | json_field_accuracy |
| violations  | array_f1            |
| explanation | llm_judge           |

# 5. `rate` 任务设计

`rate` 是内置任务，用于对文本进行多维度打分。

适用场景：

* 作文评分；
* 客服回复质量评分；
* 销售话术评分；
* 面试回答评分；
* 内容安全等级评分；
* 文案质量评价。

用户需要定义评分维度。

| 字段           | 示例        |
| ------------ | --------- |
| dimension    | clarity   |
| display_name | 清晰度       |
| description  | 表达是否清楚    |
| scale_min    | 1         |
| scale_max    | 5         |
| score_type   | integer   |
| rubric       | 1–5 分评分标准 |
| weight       | 0.3       |

`rate` 默认输出：

| 字段                   | 类型     | 说明       |
| -------------------- | ------ | -------- |
| scores               | object | 各维度分数    |
| overall_score        | number | 总分       |
| rationale            | string | 总体理由     |
| dimension_rationales | object | 各维度理由，可选 |

默认 metric：

| metric                    | 说明         |
| ------------------------- | ---------- |
| dimension_mae             | 各维度平均绝对误差  |
| overall_mae               | 总分平均绝对误差   |
| within_tolerance_accuracy | 允许一定误差的准确率 |
| exact_bucket_accuracy     | 分值完全一致比例   |
| llm_judge                 | 理由质量判断，可选  |

推荐默认值：

```text
within_tolerance_accuracy
tolerance = 1
```

因为评分任务通常存在主观波动，不建议默认使用 exact match。

# 8. LLM 调用边界

系统中有两类 LLM 调用。

## 8.1 平台级 LLM 调用

这些统一使用 OpenAI Python SDK：

| 场景                           | 说明              |
| ---------------------------- | --------------- |
| schema inference             | 根据任务描述推断输入输出结构  |
| synthetic dataset generation | 生成样例            |
| LLM-as-judge                 | 评测主观输出          |
| report summary               | 生成报告摘要          |
| data quality check           | 可选的数据质量审查       |
| prompt suggestion            | 可选的 prompt 修改建议 |

## 8.2 DSPy 内部调用

这些统一使用 DSPy：

| 场景                               | 说明                               |
| -------------------------------- | -------------------------------- |
| baseline DSPy program prediction | 基线推理                             |
| BootstrapFewShot                 | few-shot 优化                      |
| MIPROv2                          | instruction + demo 搜索            |
| GEPA                             | feedback-driven prompt evolution |
| compiled program evaluation      | 优化后评测                            |

强制规则：

```text
前端不能接触 OpenAI API key
OpenAI API key 只在后端环境变量中
DSPy optimizer service 内部不得直接 import OpenAI SDK
所有平台级 LLM 调用必须走统一 openai_client
```

# 10. 数据集设计

## 10.1 数据来源

| source_type         | 说明        |
| ------------------- | --------- |
| manual_upload       | 手动上传      |
| synthetic_generated | LLM 生成    |
| imported            | 外部导入      |
| edited              | 基于已有数据集编辑 |

---

## 10.2 数据生成流程

```text
用户输入生成命令
  ↓
系统结合 task_definition 推断 schema
  ↓
用户确认 schema
  ↓
系统生成样例计划
  ↓
分批生成数据
  ↓
结构校验
  ↓
去重
  ↓
质量打标
  ↓
写入 dataset_examples
  ↓
用户审核
```

生成样例默认状态：

```text
unchecked
```

用户审核后可以变成：

```text
accepted
rejected
needs_review
```

---

## 10.3 数据校验

每条样例需要校验：

| 检查项                     | 说明          |
| ----------------------- | ----------- |
| input_json 合法           | 能解析         |
| expected_output_json 合法 | 能解析         |
| input schema 匹配         | 必须符合 schema |
| output schema 匹配        | 必须符合 schema |
| required 字段完整           | 不缺字段        |
| enum 合法                 | 类别在允许范围内    |
| 数值范围合法                  | 评分不能越界      |
| 重复检查                    | 去重          |
| 跨 split 重复              | 阻止 run      |
| 质量状态                    | 决定能否进入 run  |

---

## 10.4 Split 策略

不同 optimizer 使用不同推荐拆分。

| 场景                  | 推荐 split                         |
| ------------------- | -------------------------------- |
| baseline evaluation | test 或 dev/test                  |
| BootstrapFewShot    | train 60%，dev 20%，test 20%       |
| MIPROv2             | train 20–40%，dev 40–60%，test 20% |
| GEPA                | train 60–70%，dev 10–20%，test 20% |

报告必须标注：

* 样例总数；
* train/dev/test 数量；
* 数据来源比例；
* accepted / unchecked / needs_review 比例；
* 是否 synthetic-only；
* 是否存在低样本风险；
* 是否存在重复或近重复。

# 11. Prompt 管理设计

## 11.1 Prompt 组成

| 部分                 | 说明           |
| ------------------ | ------------ |
| system_prompt      | 任务规则、角色、输出约束 |
| user_template      | 输入渲染模板       |
| output_schema_json | 输出结构         |
| version            | 版本号          |

Prompt 保存时必须校验：

| 校验项                    | 处理         |
| ---------------------- | ---------- |
| 模板变量不存在                | 拒绝保存       |
| output schema 非法       | 拒绝保存       |
| prompt 输出格式和 schema 冲突 | 警告或拒绝      |
| metric 字段不存在           | 创建 run 时拒绝 |
| prompt 太长              | 成本警告       |

---

## 11.2 Prompt 版本

Prompt 更新建议生成新版本，而不是覆盖旧版本。

原因：

* 历史 run 可复现；
* 可以比较不同 prompt；
* 可以回滚；
* 可以做 run comparison。

# 12. DSPy Program 设计

## 12.1 Task Adapter

系统使用 Task Adapter 将任务定义转换成 DSPy 可执行形式。

| Adapter               | 对应任务            |
| --------------------- | --------------- |
| ClassificationAdapter | classification  |
| ExtractionAdapter     | extraction      |
| QAAdapter             | qa              |
| JSONGenerationAdapter | json_generation |
| RewritingAdapter      | rewriting       |
| RateAdapter           | rate            |
| CustomTaskAdapter     | custom task_key |

Adapter 负责生成：

* instructions；
* input rendering；
* output schema；
* recommended metrics；
* report profile；
* parser rules。

---

## 12.2 Program 统一形态

第一版不让用户写 DSPy 代码。

统一使用：

```text
text -> answer
```

说明：

| 字段     | 含义                      |
| ------ | ----------------------- |
| text   | 渲染后的任务输入                |
| answer | 模型输出，通常是文本或 JSON string |

对于复杂输入，系统在 prompt 渲染阶段把多个字段合并成 `text`。

---

# 13. Metric 设计

## 13.1 默认 metric

| 任务              | 默认 metric                                     |
| --------------- | --------------------------------------------- |
| classification  | json_field_accuracy / exact_match             |
| extraction      | json_all_fields_accuracy                      |
| qa              | normalized_exact_match / token_f1 / llm_judge |
| json_generation | schema_validity + json_all_fields_accuracy    |
| rewriting       | llm_judge                                     |
| rate            | within_tolerance / MAE                        |
| custom          | 用户配置 composite metric                         |

---

## 13.2 Metric DSL

自定义任务使用受限 Metric DSL，不允许用户写 Python 代码。

支持组件：

| Metric                   | 用途      |
| ------------------------ | ------- |
| json_field_accuracy      | 单字段准确率  |
| json_all_fields_accuracy | 全字段准确率  |
| array_f1                 | 数组字段匹配  |
| numeric_mae              | 数值误差    |
| numeric_within_tolerance | 数值容差    |
| exact_match              | 完全匹配    |
| normalized_exact_match   | 归一化匹配   |
| token_f1                 | 文本 F1   |
| llm_judge                | 主观评审    |
| schema_validity          | 输出格式合法性 |

Composite metric 示例逻辑：

| 字段          | metric              | 权重  |
| ----------- | ------------------- | --- |
| risk_level  | json_field_accuracy | 0.5 |
| violations  | array_f1            | 0.3 |
| explanation | llm_judge           | 0.2 |

规则：

| 情况                | 处理        |
| ----------------- | --------- |
| metric 引用不存在字段    | 拒绝        |
| 权重总和不是 1          | 自动归一化或拒绝  |
| 数值 metric 用在字符串字段 | 拒绝        |
| LLM judge 权重过高    | 成本和非确定性警告 |
| 全部依赖 LLM judge    | 报告低确定性    |

---

## 13.3 GEPA Feedback Metric

GEPA 必须使用 feedback-rich metric。

GEPA metric 需要返回：

| 字段         | 说明       |
| ---------- | -------- |
| score      | 数值分数     |
| feedback   | 自然语言错误反馈 |
| error_type | 错误类型     |
| expected   | 期望摘要     |
| actual     | 实际摘要     |

反馈应说明：

```text
输入是什么
期望输出是什么
实际输出是什么
哪里错了
为什么错
下一版 prompt 应如何调整
```

如果 feedback 为空或只有“对/错”，不允许运行 GEPA。

# 14. Optimizer 设计

## 14.1 通用运行流程

```text
创建 optimization_run
  ↓
创建 job
  ↓
冻结 prompt / dataset / model / metric / optimizer 快照
  ↓
配置 DSPy LM
  ↓
构建 DSPy examples
  ↓
构建 DSPy program
  ↓
运行 baseline evaluation
  ↓
执行 optimizer
  ↓
运行 optimized evaluation
  ↓
保存 predictions
  ↓
保存 report
  ↓
保存 artifact
```

---

## 14.2 BootstrapFewShot

适用：

* 样例较少；
* 成本敏感；
* 快速得到 few-shot baseline。

规则：

```text
只使用 trainset
不要传 valset
max_labeled_demos 不能超过 trainset 大小
metric 全部失败时停止
```

---

## 14.3 MIPROv2

适用：

* 有较多样例；
* 希望同时优化 instruction 和 few-shot；
* 有稳定 metric；
* 可以接受更高成本。

要求：

```text
必须有 valset
num_threads 受系统限制
auto 和 num_trials 不应冲突
minibatch_size 不能大于 valset size
```

---

## 14.4 GEPA

适用：

* 错误模式复杂；
* 有失败样本；
* 需要从文本反馈中优化 prompt。

要求：

```text
必须有 feedback-rich metric
必须保存 trace log
必须设置预算限制
所有样例满分时跳过
所有样例零分时建议先修 prompt
```

---

# 15. Evaluation 和 Report

## 15.1 Baseline Evaluation

优化前必须先跑 baseline evaluation。

它检查：

* prompt 是否可渲染；
* 模型是否输出；
* 输出是否符合 schema；
* metric 是否可用；
* test set 是否有效；
* 是否值得进入 optimizer。

如果 baseline 大量解析失败，应先修 prompt，而不是直接优化。

---

## 15.2 Report 内容

报告包含：

| 模块                  | 内容                                   |
| ------------------- | ------------------------------------ |
| Summary             | baseline score、optimized score、delta |
| Dataset             | 样例数量、来源、质量状态                         |
| Metric              | metric 类型、参数、LLM judge 信息            |
| Optimizer           | optimizer 名称、参数、耗时                   |
| Score Breakdown     | 按字段、类别、维度拆解                          |
| Failed Examples     | 失败样本                                 |
| Regression Examples | 优化后变差样本                              |
| Prompt Diff         | prompt 差异                            |
| Few-shot Demos      | 被选中的 demos                           |
| Predictions         | baseline / optimized 输出              |
| Warnings            | synthetic-only、低样本、重复、schema 风险      |
| Artifacts           | 下载链接                                 |

---

## 15.3 错误类型

失败样本应分类：

| 错误类型              | 说明           |
| ----------------- | ------------ |
| parse_error       | 输出无法解析       |
| schema_error      | 不符合 schema   |
| missing_field     | 缺字段          |
| extra_field       | 多字段          |
| wrong_label       | 分类错误         |
| numeric_error     | 数值误差         |
| semantic_mismatch | 语义不符         |
| style_violation   | 风格不符         |
| empty_output      | 空输出          |
| judge_error       | LLM judge 失败 |
| gold_suspect      | 标注可能有问题      |

# 16. Worker 设计

## 16.1 状态机

| 状态               | 含义   |
| ---------------- | ---- |
| queued           | 等待执行 |
| running          | 执行中  |
| succeeded        | 成功   |
| failed           | 失败   |
| cancel_requested | 请求取消 |
| cancelled        | 已取消  |

流转：

```text
queued → running → succeeded
queued → running → failed
queued → cancelled
running → cancel_requested → cancelled
```

---

## 16.2 Claim 和恢复

Worker claim job 时：

1. 开启事务。
2. 找到 queued job。
3. 更新为 running。
4. 写入 locked_by、locked_until。
5. 定期更新 heartbeat。
6. 如果 heartbeat 超时，允许恢复或标记 failed。

---

## 16.3 取消任务

取消策略：

| 阶段                         | 处理           |
| -------------------------- | ------------ |
| queued                     | 直接取消         |
| dataset generation running | 当前 batch 后停止 |
| evaluation running         | 当前样例后停止      |
| optimization running       | 安全检查点停止      |
| artifact writing           | 写完当前文件后停止    |

不强杀进程，保留 partial artifact。

# 25. 给 Codex 的总任务说明

可以直接把下面这段交给 Codex：

```text
Build a local Prompt Optimization Studio without fine-tuning.

Backend:
- Python 3.11, FastAPI, SQLAlchemy, Alembic, SQLite.
- Use OpenAI Python SDK only for platform-level LLM calls:
  schema inference, synthetic dataset generation, LLM-as-judge, report summary.
- Use DSPy dspy.LM and dspy.configure for all DSPy internal calls.
- Support builtin tasks:
  classification, extraction, qa, json_generation, rewriting, rate.
- Support custom tasks using:
  task_kind = custom
  task_key = user-defined lowercase English identifier.
- Do not use "other" as a task type.
- Custom tasks must be configuration-based, not code-based.
- Support input/output schema, metric DSL, and schema-aware reports.
- Implement BootstrapFewShot, MIPROv2, and GEPA.
- GEPA must require feedback-rich metric.
- Store structured metadata in SQLite.
- Store large artifacts in local artifact directories.
- Implement SQLite polling worker for long-running jobs.
- Every run must snapshot prompt, dataset split, model config, optimizer config, metric config, package versions, and random seed.
- Reports must show baseline score, optimized score, delta, failed examples, regression examples, data quality warnings, and artifact links.

Frontend:
- Next.js, React, TypeScript, Tailwind, shadcn/ui.
- Pages:
  Projects, Task Definition, Dataset Generator, Dataset Import, Dataset Editor,
  Prompt Studio, Baseline Evaluation, Optimization Runs, Report, Run Comparison.

Explicitly do not implement:
- fine-tuning JSONL export
- OpenAI fine-tuning file upload
- fine-tuning job creation
- fine-tuned model storage
- fine-tuned model selection
- any fine-tuning UI
```

---

最终设计核心是：

```text
内置任务覆盖常见场景
自定义英文 task_key 覆盖扩展场景
schema 定义任务输入输出
Metric DSL 定义评测逻辑
DSPy 负责 prompt / few-shot / instruction 优化
OpenAI SDK 负责平台级 LLM 能力
SQLite 和 artifact 文件保证本地持久化与可复现
```

这套设计比简单增加一个 `other` 更稳定，也更适合后续扩展任务模板、报告模板和优化器能力。
